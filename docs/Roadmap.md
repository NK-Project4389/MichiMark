# MichiMark ロードマップ

最終更新: 2026-03-30

---

## プロダクトビジョン

> 時間 × 場所 × アクションのストーリーを記録し、欲しい情報に集計・可視化して、使う人の生産性を向上させる。

---

## マネタイズ戦略

### モデル：Freemium + サブスクリプション

| プラン | 対象 | 単価（月） | 車両台数 | 主な機能 |
|---|---|---|---|---|
| **無料** | 個人 | ¥0 | 1台 | 記録無制限・直近7日サマリー・イベントOverview |
| **個人プラン** | 個人・フリーランス | ¥300 | 1台 | 月次・任意期間サマリー・CSVエクスポート |
| **チームプラン** | 中小企業 | ¥500 / 台 | 2〜5台 | 個人プラン全機能 + 管理者ダッシュボード・車両管理 |
| **法人プラン** | 自社車両6台〜 | ¥400 / 台 | 6〜20台 | チームプラン全機能 + ボリューム割引 |

※ 20台超は運行管理システムの領域としてスコープ外
※ 年払い：2ヶ月分割引

### ターゲット戦略

| ユーザー層 | 位置づけ | 狙い |
|---|---|---|
| **訪問作業系（電気・配管・設備・クリーニング）** | 本丸・課金ターゲット | 時間記録 = 請求根拠・日報代替として毎日使う |
| **ルート営業（車移動）** | 無料ユーザー → 法人アップセル | 個人自腹は難しい。会社ごとチームプランへ誘導 |
| **旅行旅費管理** | 集客・広告代わり | 入口として取り込み、継続利用に転換 |

### スコープ定義（重要）

- **対象：自社車両の管理**（自社ドライバー × 自社車両）
- **対象外：受託・外注関係**（A社がB社の車両を管理するケースは含まない）
- **対象外：運行管理法対応**（点呼・緑ナンバー事業者向け機能は含まない）
- この制約によりOrganizationは1社完結で設計を維持し、システム複雑性を抑える

### 転換ポイント設計

```
無料ユーザーが1ヶ月使う
  → データが溜まる
  → 月次サマリーを見ようとする → 個人プラン案内（¥300転換ポイント）
  → 従業員・車両が増える → チームプラン案内（アップセルポイント）
```

## Phase 1 — 移行完了・基盤整備

**目標：SwiftUI版の機能をFlutterで完全動作させる + 将来のクラウド化・法人対応を見据えた基盤を作る**

### Flutter移行タスク

| タスク | 概要 |
|---|---|
| 設定系 Feature 完成 | trans_setting / member_setting / tag_setting / action_setting |
| EventDetail 一括保存 | 全タブの変更をまとめて保存（§17仕様） |
| Seed data 投入 | InMemory スタブにテストデータを入れて画面確認できる状態にする |
| drift Repository 実装 | 永続化レイヤーの実装 |
| get_it DI セットアップ | Repository の DI 注入基盤を整える |

### 将来対応のための設計考慮（drift実装前に対処）

| 対処 | 内容 | 理由 |
|---|---|---|
| **全エンティティIDをUUIDに変更** | auto-increment → UUID（String） | クラウド同期・マルチデバイス時のID衝突防止 |
| **Organization / Vehicle エンティティ追加** | 組織・車両のドメインモデル設計 | 後から追加するとデータマイグレーションが大変 |
| **Event に user_id / vehicle_id フィールド追加** | nullable で追加 | 法人プラン時の集計・管理に必須 |
| **Repository インターフェース抽象化** | クラウド同期を前提としたI/F設計 | ローカル専用に閉じない設計にしておく |
| **AuthContext の DI 確保** | userId / organizationId / role を DI に持たせる場所を作る | 認証レイヤー追加時のリファクタコスト削減 |

## Phase 2 — トピック機能（定型タイプ提供）

**目標：用途別にイベントの入力項目・タブ・アクション種別を最適化する**
**公開：TestFlight / 限定ベータ公開（無料）でユーザーフィードバックを収集**

### 概念モデル

```
Topic
  ├─ id / name / type（定型 or カスタム）
  ├─ 表示タブ設定（タブごとに visible: true/false）
  └─ 利用可能アクション種別
```

イベントにトピックを紐づけることで、EventDetail の表示タブが切り替わる。

### 定型トピック（Phase 2 提供予定）

| トピック | コアバリュー | 主な表示タブ |
|---|---|---|
| **作業訪問** | 移動 / 作業 / 休憩の時間記録 | basic_info, michi_info, fuel_detail |
| **営業・車移動** | 走行距離 + ガソリン代精算 | basic_info, michi_info, fuel_detail, payment_detail, payment_info |
| **旅行** | 旅費・経費の管理 | basic_info, michi_info, payment_detail, payment_info |

※ タブ構成の詳細は各トピックの要件書で定義する

### タスク

| タスク | 概要 |
|---|---|
| Topic ドメインモデル設計 | Topic エンティティ・タブ設定の設計 |
| トピック選択 UI | イベント作成時にトピックを選択する画面 |
| EventDetail タブ動的制御 | トピックに応じたタブ表示 / 非表示の切り替え |
| 定型トピック 3種 実装 | 作業訪問 / 営業・車移動 / 旅行 |

## Phase 3 — アクション × 時間モデル

**目標：アクションに状態遷移と時間計測を持たせ、フィールドワーカーの活動を記録できるようにする**
**公開：App Store 正式リリース（Freemiumモデル適用）← ファーストキャッシュの目標地点**

### 概念モデル

```
Action
  ├─ 単体型：発生したことを記録（フェーズ変化なし）
  └─ 状態遷移型：現在のフェーズを切り替えるポイント
       例）出発 → [移動中] → 到着 → [作業中] → 完了

Phase（状態）
  ├─ リアルタイムタイマー（開始 / 終了）
  └─ サブアクション（トグル型）
       例）作業中フェーズ内の「休憩」トグル
           ON = 休憩開始 / OFF = 休憩終了
           ※ フェーズ自体は「作業中」のまま継続
```

### タスク

| タスク | 概要 |
|---|---|
| Action ドメインモデル拡張 | 状態遷移型 / 単体型の設計 |
| Phase（状態）モデル設計 | フェーズ管理・タイマーの設計 |
| サブアクション（トグル）設計 | フェーズ内ネストアクションの設計 |
| リアルタイムタイマー UI | 計測中の表示・開始 / 停止操作 |
| 場所の手動登録 | アクション発生時の場所を手動入力（GPS は Phase 5 以降） |
| イベント Overview 画面（先行実装） | 1件のイベント内の時間・距離・費用サマリー |
