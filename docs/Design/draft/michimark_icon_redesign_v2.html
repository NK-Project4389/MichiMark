RATIONALE

デザイン根拠（v2 追加・変更分）

v1 の根拠（テキスト除去・ピンのソリッド化・グラデーション排除等）は引き続き有効。v2 で変更した判断のみを記録する。

v2 でピンを大きくした理由

v1 のピンは小サイズで「ピンの尖り」が消えてしまい、白い菱形に見えてしまう課題があった。v2 では頭部（膨らみ）をより大きく取り、ピン先が尖って見える形を維持。130px 時点でピン形状が明確に読める設計にした。

Plan C に深緑を採用した理由

ブルー系3案を並べると差別化が弱く「どれを選んでも同じ印象」になりやすい。Plan C は計測・記録アプリとしての独自性を最前面に出す案なので、大地・道・自然を想起させる深緑（#1A6B47）で差別化を図った。現行ブランドから離れるリスクはあるが、提案として幅を持たせた。

各案にスプラッシュを対応させた理由

v1 はアイコン3案 + スプラッシュ1案（Plan B対応）という構成だったため、「アイコンA案を選んだらスプラッシュはどうなる？」という疑問が残った。v2 では各案が一貫したブランドイメージで揃うよう、スプラッシュも3案並べて提示した。

WCAG コントラスト比確認（v2）

  • 白 on #1B6D8E（Plan A）: 約 4.9:1（AA 合格）
  • 白 on #0F4A62（Plan B）: 約 8.2:1（AAA 合格）
  • 白 on #1A6B47（Plan C）: 約 5.1:1（AA 合格）
  • アンバー #E8A020 on Deep: 約 5.6:1（AA 合格）
  • アンバー #E8A020 on Forest: 約 3.4:1（AA Large 合格）

4サイズプレビューを追加した理由

v1 は 120 / 60 / 40px の3サイズだった。iOS の通知アイコン・Spotlight サイズ（29px前後）での見え方を確認するために 29px を追加。特に Plan C は目盛りが潰れるため、小サイズではルーラー帯の存在感だけで識別させる設計になっていることを確認した。

アイコンにアンバーを入れなかった理由

アンバー（#E8A020）はアクセントポイントとして有効だが、アイコン全体に使うとカジュアル・派手な印象になりビジネスユーザー向けの信頼感が下がる。スプラッシュでのアンバードットは「計測ポイント」という機能的文脈があるため許容。Plan B アイコンのみ1点の小さなドットとして使用している。

NEXT STEPS

提案後の進め方

  1. ユーザーが アイコン案（A / B / C / ブレンド）とオプション3項目 をフィードバック
  2. designer が必要に応じて 修正案または v3 を作成
  3. 採用案確定 → product-manager が 要件書（docs/Requirements/REQ-BRAND-icon-splash.md） に整形
  4. 要件書をもとに Figma データ作成（本 HTML の SVG をベースに 1024x1024px でトレース）
  5. Figma から 1024x1024 PNG エクスポート → flutter_launcher_icons / flutter_native_splash で各解像度を自動生成
  6. スプラッシュアニメーションは 別フェーズ（要件書・Spec 分離）で実装
