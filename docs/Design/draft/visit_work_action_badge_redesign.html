01 / Problem Analysis

現状の課題

課題A: アクションバッジが小さすぎる

状態バッジ（_ActionStateBadge）は fontSize:10・padding vertical:2 の極小サイズ。タップターゲットとして機能しておらず、状態変更の操作感が弱い。

課題B: 削除ボタンとアクションボタンの距離が近すぎる

⚡アクションボタン（28px）と削除ボタン（36px）の間隔が left:4dp のみ。カード高さ72px に対して誤タップを誘発しやすいレイアウト。

課題C: 削除ボタンの視認性が高すぎる

赤い削除ボタンがカード右端に常時表示されており、ユーザーが誤って触れるリスクがある。削除は頻繁に行う操作ではないため、常時露出は不適切。

課題D: タップ操作の混在

カードタップ（詳細遷移）・⚡ボタンタップ（ActionTime起動）・削除ボタンタップが72pxの高さに集中しており、誤操作のリスクが高い。

07 / Rationale

デザイン根拠・判断理由

📏

タップターゲットサイズの基準

Apple HIG および WCAG では最小タップターゲットを 44x44dp 以上と推奨。現状の⚡ボタン（28px）・状態バッジ（事実上タップ不可）はいずれも基準を下回っており、特に訪問作業中の片手操作・移動中の操作で誤タップリスクが高い。36x36dp の⚡ボタンへの拡大と、状態バッジのタップ機能付与で実用的なタップエリアを確保する。

🔒

削除操作は「意図的に発動する」べき

削除はデータを失う不可逆操作であり、常時ボタン表示は不要。iOSのメール・リマインダー・連絡先など標準アプリはすべてスワイプまたは長押しで削除を発動する。MichiMark も同じパターンに揃えることで、誤削除防止と操作一貫性を同時に達成できる。

🔄

状態バッジをインタラクティブ要素に昇格

現状の状態バッジは読み取り専用で、ActionTimeの起動は隣の⚡ボタンから行う必要がある。バッジ自体をタップ可能にすることで、「状態を見てそのまま変更できる」という直感的なフローが生まれ、⚡ボタンとの役割の重複も解消される。将来的にはバッジタップでステートシートを開くなどの拡張も可能。

🚗

MichiMark らしさ — 移動中・片手操作を前提とした設計

訪問作業トピックの主要ユーザーは車移動中・作業現場での操作が多い。大きく押しやすいボタン・間違えにくいレイアウトは UX として最優先事項。小さなバッジや密集ボタン配置は「デスクトップ前提」の設計であり、フィールドユーザーには不向き。

⚖️

コントラスト比（WCAG AA 基準）

改善後の状態バッジ: 白背景上の #7C3AED テキストはコントラスト比 7.9:1（WCAG AA: 4.5:1 以上を満たす）。削除スワイプアクション: 赤背景 #DC2626 上の白テキストはコントラスト比 5.4:1（AA 基準クリア）。

product-manager への確認事項

削除方式（案A スワイプ / 案B 長押し / 案C 下部配置）のどれを採用するかはユーザーの操作習慣・実装工数・テスト容易性とのトレードオフを踏まえ、要件書で方針を確定してほしい。 デザインとしては案A（スワイプ）を推奨するが、Integration Testでのスワイプ操作のテスト難易度も考慮すること。 また、状態バッジのタップ機能（⚡ボタンとの二重トリガー）についても、既存の ActionTime BLoC とのインターフェースに影響がないか architect と確認が必要。
