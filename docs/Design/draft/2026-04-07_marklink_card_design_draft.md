## v4.0 追記: カード間挿入ボタン 提案（2026-04-07）

### 追加要件の背景

タイムライン上のMarkとLinkの間に「＋」ボタンを設置し、カードを途中挿入できる機能を追加したい。

### 挿入ポイント

```
● [Markカード]
   ← ここに「＋」（Mark後、Link前）
□ [Linkカード]  50km
   ← ここに「＋」（Link後、Mark前）
● [Markカード]
   ← ここに「＋」（末尾追加）
```

### 採用カラー

| 役割 | HEX | 用途 |
|---|---|---|
| Insert Amber | `#F59E0B` | 挿入ボタン背景・枠・インジケーター |
| Insert Amber Dark | `#D97706` | ホバー・グラデーション終点・ヒントテキスト |
| Insert Amber Tint | `#FEF3C7` | ボタン周辺背景（使用する場合） |

Amber採用理由: Mark（Teal）・Link（Emerald）と明確に色区別でき「追加アクション」として独立した役割を持てる。「白抜き＋Teal枠」はMarkドットと混同されるため不推奨。

### 提案パターン概要

| パターン | 方式 | 推奨度 |
|---|---|---|
| **Pattern 1: 常時表示型** | 縦線上に小さな＋ボタンを常時配置 | 短期フォールバック候補 |
| **Pattern 2: タップ展開型** | カードタップで下部に＋が出現（アニメーション） | 将来拡張向け |
| **Pattern 3: FAB型** | 画面右下FAB＋挿入モード時インジケーター | **推奨** |

### 推奨案: Pattern 3（FAB型）× ボトムシート種別選択

#### 理由
1. 閲覧時にタイムラインを一切汚さず、C-2スタイルのクリーンなビジュアルを維持できる
2. FABタップで挿入モードに切り替わる明確なUXフローがiOS/Android慣習と整合する
3. 挿入ポイント（どこに）と挿入種別（何を）を2ステップに分離し、各UIをシンプルに保てる
4. 末尾追加・中間挿入を同一FABフローで統一できる

#### 短期フォールバック
Pattern 1（常時表示型）は実装コストが最も低く、初期リリース段階での採用に向く。UXが安定後にPattern 3へ段階移行を推奨。

### 挿入種別選択UI — ボトムシート仕様

- 表示トリガー: 挿入ポイントタップ後
- 内容: 「Mark を挿入（地点・スポット）」と「Link を挿入（区間・道のり）」の2択
- スタイル: ボトムシート / ハンドルあり / 2カラムグリッド
- Mark側: Teal系背景（#EAF5FB） / Teal枠（#D0EBF5）
- Link側: Emerald系背景（#EDFAF4） / Emerald枠（#D4F0E4）
- 挿入できる種別に制約がある場合（例: Mark後にはLinkのみ許可）はグレーアウト表現

### 挿入ポイントインジケーター仕様（挿入モード時）

- 表示: 挿入モード中のみ（FABタップ後）
- 形状: 小ドット（8dp / Amber 50% opacity / Amber枠） + 点線（Amber 20% opacity）
- テキスト: 「挿入ポイント」「末尾に追加」（9px / W600 / Amber 70% opacity）
- 非選択時: 表示なし（タイムラインはクリーン）

### 変更項目（v4.0 追加分）

- [ ] 挿入ボタンカラー定義: Amber `#F59E0B` / Amber Dark `#D97706` / Amber Tint `#FEF3C7`
- [ ] Pattern 3 FABボタン: 44dp円形 / Amberグラデーション背景 / 白＋記号 / shadow
- [ ] 挿入モード中インジケーター: 各カード間に小ドット＋点線＋テキスト表示
- [ ] ボトムシート: Mark挿入 / Link挿入 の2択 / ハンドルあり
- [ ] Pattern 1 常時表示ボタン（フォールバック）: 20dp円形 / 白背景 / Amber枠 / ホバーで塗りつぶし
- [ ] 挿入種別に制約がある場合のグレーアウト表現

## v3.0 追記: Linkカードコンパクト化 再提案（2026-04-07）

### ユーザーフィードバック

> 「リンクカード"だけ"を短くして、短くした分の余白に距離を持ってくるとだいぶイメージが合ってくると思う」

### v3.0 設計方針

- **Markカード**: 高さ維持（64dp）
- **Linkカード**: 高さを縮小してコンパクト化
- **距離ラベル**: Linkカードを縮小して生まれた余白スペースに配置（カード外）

### 3バリエーション概要

| バリアント | Linkカード高さ | 距離配置 | 推奨度 |
|---|---|---|---|
| **V1: コンパクト中 + 上余白バッジ** | 42dp（Mark比65%） | Linkカード上・縦線右に矢印バッジ | サブ候補 |
| **V2: 超コンパクト + 右端距離数値** | 34dp（Mark比53%） | Linkカード行の右外に数値+単位 | **推奨候補** |
| **V3: バーライン + 縦線沿い距離テキスト** | 26dp（Mark比40%） | 縦線右に点線+テキスト（沿道標識風） | 将来拡張向け |

### V2 を推奨候補とする理由

1. フィードバックの「余白に距離を持ってくる」に最も直接的に応答している
2. Linkカード行（Row）の右端に距離数値を独立配置するシンプルな構成で、実装コストが低い
3. 距離が null の場合のフォールバックが容易（数値エリアを空にするだけ）
4. 大きめの数値（13px W800 / Emerald色）で距離の視認性が最も高い
5. MarkカードとLinkカードの高さ差（64dp vs 34dp）がタイムラインの視覚的リズムを最大化

### V1・V3 の位置づけ

- **V1**: 距離を「バッジ」として縦線上に浮かせたい場合の代替案。v2.0 の C-2 に近い思想を継承。
- **V3**: Linkカードを「バー」として最もミニマルに扱う場合。将来の「タップで展開」実装との相性が良い。

## 概要

MichiInfo タイムライン画面に表示される Mark（地点）カードと Link（区間）カードのデザインを定義する。
デザインレポートで提案した4パターン（A: ミニマル / B: ビジュアル重視 / C: タイムライン連結 / D: ダークモード）のうち、
**パターン C（タイムライン連結型）を推奨案とする**。

さらに、パターンCのバリエーション深掘りにより **C-2（中央フローティングバッジ型）を推奨バリエーション** として選定した。

---

## 重要制約: 距離はカード外に表示する

**距離情報はリンクカード内に表示しない。**

### 理由

カード内に距離を配置すると「Linkカードの属性情報」として誤認される可能性がある。
距離はあくまで Mark 間の移動を示す情報であり、タイムラインの「線」部分に紐づく情報として表現する。

### 採用する配置方式（C-2推奨）

Link 区間縦線の中央に距離バッジ（小ラベル）を浮かせて配置する。

- 位置: Link 縦線の中央・左寄り（縦線に重なるように配置）
- スタイル: 9px / W700 / 白文字 / Emerald背景(#2E9E6B) / 角丸6px / shadow
- 縦線幅: 6px（グラデーション #2E9E6B → #1A7A52）により距離バッジとの一体感を演出

---

## 推奨パターン: C-2 タイムライン連結型・中央フローティングバッジ

### 推奨理由

1. MichiMark のコアコンセプト「道を記録する」を縦線・ドット・カードの連結で直感的に表現できる
2. 距離情報が「線の情報」であることが最も自然に伝わる配置（カード外・縦線中央バッジ）
3. 既存の REQ-michi_info_layout.md が定義した「縦線 + ドット + 接続」設計と方向性が一致する
4. REQ-michi_info_timeline_redesign.md の「水平罫線接続」「縦線短縮」要件とも自然に組み合わせられる
5. Mark（円ドット）と Link（角丸矩形ドット）を形状でも差別化し、色のみに依存しない視覚区別を実現する

## バリエーション比較（C-2推奨の根拠）

| バリエーション | 距離配置 | 特徴 | 推奨度 |
|---|---|---|---|
| C-1: サイドラベル | 縦線左・縦書きテキスト | シンプル・実装コスト低 | サブ候補 |
| **C-2: 中央バッジ** | **Link縦線中央・浮かせバッジ** | **直感的・可読性高・バランス最良** | **推奨** |
| C-3: セパレータ | 独立セパレータ行として挿入 | 距離の独立性最高・点線縦線 | 密なリストでは重い |
| C-4: 沿道テキスト | Link縦線右脇・水平テキスト | 道路メタファー強・縦線幅広 | 実装コスト高め |

## PMへのメモ

- C-2 の「距離バッジ浮かせ配置」は Flutter の Stack + Positioned で実現可能と想定。architect / flutter-dev の実装コスト確認を推奨。
- 距離バッジがない（距離データが null）場合のフォールバック表示についても要件に含めると良い。
- C-3（点線縦線 + セパレータ行）は「距離が目立つことを優先する場合」の代替として残しておく価値がある。
- パターン A を先に実装してからパターン C-2 に段階移行するアプローチも引き続き有効。
- ダークモード（パターン D）は現行スコープ外。別要件として切り出し推奨。
