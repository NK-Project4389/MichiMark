C案 レイアウト詳細

左1列（C・⌫・.）を「補助キー列」とし、数字3列（7-9, 4-6, 1-3）との間に自然な視覚グループを形成。 右端列は演算子専用（÷ × − +）。最下段は「00 / 0 / +（ラスト演算子） / 確定（全幅）」の2段構成。 確定ボタンを全幅の独立行にすることで、入力完了アクションの視認性と誤打防止を両立。

長所

  • 確定ボタンが全幅・独立行のため絶対に見逃さない。誤打もゼロに近い
  • 補助キー列（C・⌫・.）が左端に集約され、親指操作で自然に届く
  • 演算子が専用列（右端）に縦並びで、電卓モードの操作フローが完結
  • isDecimal=false 時は「.」を非活性グレーにするだけで直感的にモードが伝わる
  • キーの縦横比・面積が均等で、タップターゲットが全機種で十分な大きさ

短所

  • 5列のため iPhone SE（375pt）では各キー幅が約65ptとやや狭い（許容範囲内）
  • 「確定」全幅行があるため全体の高さが他案より1行分高くなる
  • iOS標準電卓との配置差が最も大きく、ユーザーによっては最初に戸惑う可能性

6. 表示モード（Push-up vs Overlay）

Push-up 方式（推奨）

システムキーボードと同様に画面コンテンツを上方向に押し上げ、入力中のフィールドを常に可視領域に保つ方式。

推奨理由: NumericInputRow は画面の中央〜下部に配置されることが多い。 Overlay 方式だとキーパッドに隠れて入力中の値が見えなくなるリスクがある。 Push-up なら常に「ラベル + 現在値 + 単位」が見える状態で入力できる。

注意点: Scaffold の resizeToAvoidBottomInset: true に相当する挙動を カスタムキーパッドでも再現する必要がある（AnimatedPadding 等での制御が必要）。

Overlay（モーダルシート）方式

画面上にオーバーレイとして表示する方式。BottomSheet API の使用が想定される。

長所: 既存画面のレイアウト変更が不要。実装コストが低い。

短所: キーパッドがコンテンツ上に被さるため、入力中の値が隠れやすい。 ただし本デザインではキーパッド上部の Display エリアに値を常時表示しているため、 フィールド自体が隠れても運用上は問題ない。セカンダリ候補として有効。

設計指針

  • キーパッドの Display エリアに「ラベル名・単位・変更前値・現在値」を集約することで、Push-up でも Overlay でもユーザーは入力情報を失わない
  • まず Overlay（BottomSheet）で実装してリリースし、UX フィードバックを受けて Push-up 方式に移行するロードマップを推奨

7. 根拠・設計判断

1 カスタムキーパッドを採用する理由

システムキーボード（TextInputType.number）はデバイス・OS バージョンによってレイアウトが異なり、 「00」「演算子」「変更前値の表示」を統一的に提供できない。 カスタムキーパッドにすることでキー配置・演算機能・Draft 表示を完全にコントロールできる。

2 演算子機能の理由

ガソリン給油量（走行距離 ÷ 燃費）や支払金額（単価 × リットル数）など、 ドライブ記録では簡易計算ニーズが発生しやすい。 アプリを切り替えずにその場で計算できることで UX が向上する。

3 Draft original の表示理由

編集モードで既存値を変更する際、「元の値がいくつだったか」を確認しながら入力できる。 特に燃費・給油量などは誤入力訂正のユースケースが多い。 変更前値を薄く1行目に表示することでコンテキストを保持しつつ入力に集中できる。

4 isDecimal 連動 (. キー非活性)

整数フィールド（走行距離・金額）では小数点入力を防ぐため「.」を視覚的に非活性化する。 色を薄くするだけで済むため実装コストは最小限。 「00」キーは金額系フィールドで特に有用（5,400円 などを素早く入力可能）。

5 Teal アクセントの設計判断

確定（=）キーを Teal Dark (#2D6A6A) で塗ることで、 画面内で最もコントラストの高い「完了アクション」として視線を誘導。 演算子は Teal Tint で「サブアクション」として階層化し、 数字キーは純白（ライト）/ダークグレー（ダーク）でニュートラルに保つ。

6 アスペクト比固定・等比スケールの理由

iPhone SE〜Pro Max の幅差は 375〜430pt（約 15%）。 固定ピクセルではキー高が小さすぎたり大きすぎたりするため、 MediaQuery.size.width 比率でスケールする設計を採用。 縦横比を固定することで機種間での操作感の一貫性を担保する。

8. 推奨案

C案（5列ハイブリッド型）を推奨

MichiMark の数値入力ユースケース（精度が求められる燃費・金額・距離）において、 「確実な確定操作」「誤タップ防止」「演算子との棲み分け」の3点が最も優れているため。

  • 確定ボタンを全幅・独立行にしたことで、タップ操作の終着点が明確
  • C・⌫・. を左端補助列に集約したことで、数字入力エリアが汚れない
  • シンプルな色分け（白 → teal淡 → teal濃）でキーの役割が一目でわかる
  • Display エリアの Draft 表示 + 現在値の2行構成は全案共通で採用
