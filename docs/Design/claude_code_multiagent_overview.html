Claude Code 活用事例

マルチエージェント設計で
アプリ開発してみた

専門特化した8つのAIエージェントに役割を分担させ、
設計 → 実装 → テスト → レビューを自動化するパターン。

Multi-Agent

各エージェントは
.claude/agents/
のMDファイルで定義
8つのエージェント
🎯
Product Manager
Requirements
要望を受け取り要件書に落とし込む。開発の起点。
🏗️
Architect
Spec Design
要件を設計仕様（Spec）に変換。テストシナリオまで定義。実装はしない。
💻
Developer
Implementation
Specだけを見てコードを書く。Spec不足なら実装を止める。
🧪
Tester
Test + Execution
実装コードを見ずSpecのシナリオだけでテストを書く（ブラックボックス）。
🔍
Reviewer
Code Review
実装＋テスト両方揃ってからレビュー開始。コードは書かない。
🎨
Designer
UI Proposal
UIデザインをHTMLで提案。要件の叩きをPMに渡す。
🎼
Orchestrator
Coordination
進捗・タスクボード管理。テスト実行時の監視も担当。
📋
Charter Reviewer
Architecture Docs
設計ドキュメントのレビューと改善提案。

その他のフロー
🐛 バグ修正サイクル
Developer ⚡並行 Tester → Reviewer → Tester実行 → Push
バグ修正後も必ずTesterがテストを書いて実行する
🎨 デザイン提案フロー
Designer → PM → 承認 → Architect → Developer → Push
DesignerはHTMLで提案。ユーザー承認なしに実装に入らない
このアプローチのポイント
📄
Spec駆動で迷わない
全機能を先にSpecで定義。コードから仕様を推測しない。Specがゲートキーパー。
🔒
役割の越権を禁止
Architectは実装しない、Reviewerはコードを書かない。境界を守ることで品質が安定する。
🧪
テストは省略不可
全件PASS前にPushしない。FAILが出たらコードを直してからサイクルを回す。
📋
タスクボードで競合防止
複数セッション並行時にIN_PROGRESSのタスクに別セッションが触れない仕組み。
🔀
並行で高速化
実装コードとテストコードは独立して同時進行。テスト実行もシミュレーター2台で分割。
📂
MDファイルで定義
.claude/agents/ にロールごとのMDを置くだけ。Gitで管理・チームで共有できる。
